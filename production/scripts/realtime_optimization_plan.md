# 实盘交易引擎性能与健壮性优化计划 (Real-time Optimization Plan)

针对 IBKR 实盘环境中的“四大隐形杀手”（协程风暴、API 限流、流动性陷阱、状态分叉），本计划制定了从“单体异步”向“防御性微服务”进化的四阶段实施方案。

---

## 🛑 核心风险分析

1.  **Event Loop Starvation (协程风暴)**：大数据解析占用 CPU 导致订单监控协程苏醒过晚。
2.  **API Rate Limiting (接口限流)**：短时间内高频拆单触发券商 Gateway 熔断。
3.  **Liquidity Mirages (流动性幻象)**：基于盘口深度的静态假设被做市商算法狙击。
4.  **Silent State Divergence (状态分叉)**：网络抖动导致本地账本与券商持仓“鸡同鸭讲”。

---

## 🛠️ 四大优化方案实施细节

### 方案一：剥离计算与执行 (信号与执行解耦)
**目标**：彻底解决 CPU 密集型任务（Alpha 计算）阻塞 I/O 密集型任务（订单监控）的问题。
- **实施步骤**：
  - [ ] **Signal Engine**：将 `V8Orchestrator` 的 Alpha 计算逻辑剥离，仅负责接收 Redis K 线、跑模型、写入 `TRADE_SIGNAL`。
  - [ ] **OMS (Execution Engine)**：新建独立的订单管理进程，监听 `TRADE_SIGNAL`，专用 Event Loop 与 IBKR 通信。
  - [ ] **共享总线**：通过 Redis Pub/Sub 或 Stream 进行跨进程通信。

### 方案二：引入“真实状态对账” (Reconciliation)
**目标**：建立以券商数据为准的底层真相 (Ground Truth)，消除内存幽灵状态。
- **实施步骤**：
  - [ ] **后台核数线程**：启动独立线程（非 async loop）每 5-10 秒执行一次 `reqPositions()`。
  - [ ] **差异熔断逻辑**：对比 `AccountingModule.open_positions` 与 IBKR 真实持仓。
  - [ ] **处理策略**：若 `abs(local_qty - real_qty) > 0`，立刻冻结该标的所有新发单，并尝试进行自动化对冲补全。

### 方案三：动态冰山与自适应滑点 (Smart Routing)
**目标**：防御做市商的抽单行为，提高大单成交质量。
- **实施步骤**：
  - [ ] **试水机制 (Water Testing)**：单笔冰山首单仅发送总体量的 10-20%。
  - [ ] **流动性探测**：根据首单成交速度和盘口撤单反应（Spread 扩张率）动态调整剩余订单的推进速度及其单次体量。
  - [ ] **TTL 动态缩容**：若探测到流动性瞬间枯竭，立刻缩小下一环冰山的 Limit Price 偏差。

### 方案四：并发协程限额 (Bounded Semaphores)
**目标**：防止极端行情下高并发任务挤兑系统资源。
- **实施步骤**：
  - [ ] **并发锁管理**：在 `OrchestratorExecution` 中引入 `asyncio.Semaphore(value=10)`。
  - [ ] **任务队列**：确保同时处于监控状态的订单上限受控，保护 IBKR API 的请求限额，防止吞吐量瞬间溢出导致被限流。
