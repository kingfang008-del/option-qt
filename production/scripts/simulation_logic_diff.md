# Backtest vs. LIVEREPLAY/Realtime 仿真差异与影响分析

本文件详细对比了 `backtest`（回测模型）与 `LIVEREPLAY`（离线重放仿真，及所有 `realtime` 模式）在核心逻辑上的本质差异，并提供了一套测试计划以评估在高并发数据流下性能衰减对执行一致性的致命影响。

---

## 核心技术差异对比表

| 维度 | Backtest 模式 (离线建模) | LIVEREPLAY / Realtime 模式 (生产仿真) | 对大型数据重放的致命影响 |
| :--- | :--- | :--- | :--- |
| **成交与滑点模型** | **静态/瞬时**。使用 `SLIPPAGE_ENTRY_PCT` 直接计算成交价，无延迟、不拆单。 | **动态/流式**。进入 `_execute_entry` 复杂的下半部分，涉及冰山拆单 (`_iceberg_open_order`) 和限价单监控。 | 在高频 Tick 下，冰山策略的拆单计算会产生显著累计延迟。 |
| **订单监控机制** | **无监控**。基于 Bar 假设成交，跳过所有异步任务。 | **异步协程监控**。启动 `asyncio` 协程模拟挂单、撤单、追单。代码路径与生产 100% 一致。 | 大量挂单会产生数千个 `asyncio` 任务，在高并发瞬间可能导致事件循环 (Event Loop) 阻塞。 |
| **PnL 监控频率** | **结算触发**。仅在平仓时统计，不运行后台监视器。 | **帧同步触发**。在 `process_batch` 帧末尾同步触发 PnL 检查。 | 虽然保证了帧对齐，但增加了一次全局持仓扫描的 CPU 开销。 |
| **状态恢复与预热** | **SQLite 快速加载**。直接读取历史记录填充缓存。 | **生产级恢复**。模拟从 PostgreSQL 恢复，或通过专门的 SQLite 恢复逻辑还原生产级状态。 | 启动开销增大，状态恢复失败可能导致首帧交易逻辑异常。 |
| **帧同步与流控** | **无锁流式读取**。追求极致的计算速度，不等待。 | **Redis 强同步 (Run-Step)**。通过 `sync:orch_done` 锁进行强同步，处理完一帧才允许下一帧。 | **致命**：如果逻辑处理耗时超过 1s，会导致实盘驱动程序 (Driver) 的消息堆积。 |

---

## 高并发/大流量下的致命影响风险

当 **关闭同步模式 (`SYNC_EXECUTION = False`)** 并在高频数据（如 1s 逐笔期权行情）下运行 `LIVEREPLAY` 时，可能出现以下风险：

1. **响应滞后 (Lag)**：生产逻辑中的异步监控若未能及时处理完旧订单，新行情已经灌入。这会导致“下单 -> 监控成交 -> 下下一单”的连锁反应出现时间上的错位。
2. **状态分叉 (State Divergence)**：离线回测认为你以 $3.00 成交了，但 `LIVEREPLAY` 在模拟追单过程中，因为行情闪跳，实际可能以 $3.05 成交，甚至撤单失败。
3. **CPU 激增 (Event Loop Starvation)**：同时对 50+ 标的进行冰山拆单和挂单监控时，Python 的 `asyncio` 可能成为瓶颈，导致处理一帧的时间远超 1 秒。

## 阶段性对比测试计划 (Impact Analysis Plan)

为了量化上述逻辑差异带来的影响，建议按照以下步骤逐个开启/关闭功能进行对比测试：

### 第一阶段：基准对齐测试 (Baseline)
- **设置**：`SYNC_EXECUTION = True`, `DISABLE_ICEBERG = True`。
- **目标**：验证在此配置下，`LIVEREPLAY` 的 PnL 曲线是否与 Backtest 100% 重合。
- **指标**：PnL 差异, 总交易笔数。

### 第二阶段：执行语义测试 (Slippage/Iceberg Impact)
- **设置**：`SYNC_EXECUTION = True`, `DISABLE_ICEBERG = False`。
- **目标**：观察“异步/冰山拆单”逻辑在同步环境下的表现。
- **观察点**：成交价偏差、成交耗时（挂单到完全成交的时间跨度）。

### 第三阶段：性能压力测试 (Async/Concurrency Impact)
- **设置**：`SYNC_EXECUTION = False`, 使用 1s 逐笔数据。
- **目标**：观察完全生产代码路径下的处理延迟。
- **核心指标**：使用 `time.perf_counter` 记录 `process_batch` 的平均耗时，特别是当 `active_count`（持仓数）超过 10 时的耗时增长情况。

### 第四阶段：稳定性溢出测试 (Full Load)
- **设置**：全量 12 标的同步高频运行，不使用 Redis 锁限制 Driver。
- **观察点**：内存占用增长（协程泄漏检查）、网络 I/O 队列堆积。

---

> [!CAUTION]
> **结论**：Backtest 追求极致的建模速度，LIVEREPLAY 追求极致的仿真。在验证“实盘如何拆单”时必须用 LIVEREPLAY，但在追求海量历史数据统计性能时，应优先使用 SYNC 模式对齐 Backtest。
